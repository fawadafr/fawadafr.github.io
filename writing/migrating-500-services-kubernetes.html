Migrating 500 Services to Kubernetes

November 10, 2025 • 12 min read

In early 2024, we faced a critical decision: continue managing our monolithic deployment infrastructure or migrate 500+ microservices to Kubernetes. The existing system was straining under the weight of explosive growth—10M+ daily active users, hundreds of deployments per day, and an infrastructure bill that was growing faster than revenue.

This is the story of how we orchestrated a zero-downtime migration to Kubernetes over 9 months, the hard lessons we learned, and the patterns that actually worked at scale.

The Challenge: 500 Services, Zero Downtime

Our existing infrastructure was a mix of AWS EC2 instances managed through a custom deployment system built over 5 years. It worked, but barely. Each deploy required manual coordination, rollbacks were risky, and we had no consistent way to manage configuration across environments.

Initial Constraints

  • Timeline: 9 months from planning to full migration
  • Team Size: 12 engineers (6 platform, 6 SRE)
  • Services: 500+ microservices across 4 major product lines
  • Traffic: 10M+ DAU, 2M peak concurrent users
  • Uptime SLA: 99.95% (no more than 4 minutes downtime/month)

The zero-downtime requirement wasn't negotiable. Any service disruption would directly impact revenue and user trust. We needed a migration strategy that allowed us to move incrementally while maintaining production stability.

Phase 1: Building the Foundation (Months 1-3)

We started with the fundamentals: cluster design, networking, and tooling. The goal was to have a production-ready Kubernetes foundation before migrating any services.

Cluster Architecture Decisions

After evaluating several approaches, we settled on a multi-cluster architecture:

  • Production Clusters: Dedicated EKS clusters per region (us-east-1, us-west-2, eu-west-1)
  • Staging Cluster: Single cluster for all pre-production testing
  • Dev Cluster: Shared cluster for developer workloads
  • Management Cluster: Runs cluster operations tooling (ArgoCD, monitoring, etc.)

The multi-cluster approach added operational complexity but gave us clear blast radius boundaries. A bad deploy in us-west-2 wouldn't cascade to other regions.

Networking Strategy

We used AWS VPC CNI for pod networking with:

  • Pod security groups for fine-grained network policies
  • Private subnets for all worker nodes
  • AWS Load Balancer Controller for ingress
  • VPN peering between clusters for cross-region communication

Phase 2: Service Migration Pattern (Months 4-7)

With the foundation in place, we developed a repeatable migration pattern. The key insight: migrate services in waves, starting with the lowest-risk services first.

Migration Waves

  • Wave 1 (Internal Services): Background workers, batch jobs - 50 services
  • Wave 2 (Low-Traffic APIs): Admin APIs, analytics endpoints - 75 services
  • Wave 3 (Medium-Traffic APIs): Secondary features - 150 services
  • Wave 4 (Critical APIs): Core user-facing services - 125 services
  • Wave 5 (Data Services): Stateful services, databases - 100 services

The Migration Playbook

For each service, we followed this pattern:

  1. Containerize: Create Dockerfile, optimize for size and security
  2. Helm Chart: Standardized chart with our best practices baked in
  3. CI/CD Update: Deploy to dev → staging → canary → production
  4. Traffic Split: Gradually shift traffic from EC2 to K8s (10% → 50% → 100%)
  5. Monitor & Validate: 7-day observation period before decommissioning EC2

Phase 3: Operational Excellence (Months 8-9)

The final phase focused on making Kubernetes production-grade: monitoring, cost optimization, disaster recovery, and developer experience.

Key Improvements

  • Observability: Prometheus + Grafana + Loki for metrics, logs, traces
  • Cost Optimization: Cluster Autoscaler, spot instances, right-sizing
  • Security: Pod Security Standards, OPA policies, secret rotation
  • Developer Experience: Telepresence for local dev, kubectl plugins, documentation

Results & Lessons Learned

After 9 months, we successfully migrated all 500 services to Kubernetes with impressive results:

Outcomes

  • Uptime: 99.98% during migration (exceeded our SLA)
  • Infrastructure Costs: 35% reduction through better resource utilization
  • Deploy Frequency: 5x increase (200 → 1000+ deploys/week)
  • Deploy Time: 80% reduction (45 min → 9 min average)
  • MTTR: 60% improvement (30 min → 12 min average)

Hard Lessons

1. Start with stateless services. We learned this the hard way. Our first attempt to migrate a stateful service (PostgreSQL replica) failed spectacularly. Stick to stateless workloads until you have battle-tested patterns for stateful sets.

2. Networking is harder than you think. DNS propagation, load balancer warmup, connection draining—all of these caused issues we didn't anticipate. Build extra time for networking debugging.

3. Developer experience matters more than perfection. We initially built a "perfect" Helm chart with every possible configuration option. Developers hated it. The simplified version with sane defaults got 90% adoption in weeks.

4. Cost optimization is not optional. Our first month's K8s bill was 40% higher than EC2. Autoscaling, spot instances, and resource right-sizing brought it down to 35% less than EC2.

Would We Do It Again?

Absolutely. The migration was painful, but the operational improvements have been transformative. Our engineering teams ship faster, our infrastructure is more reliable, and our costs are lower.

If you're considering a similar migration, my advice: start small, move incrementally, and prioritize developer experience from day one.
